# Getting Started with Ferrum

This guide walks you through integrating Ferrum into a new .NET MAUI app that
calls into a C/C++ static library.

## Prerequisites

| Tool | Minimum version |
|---|---|
| .NET SDK | 9.0 |
| .NET MAUI workload | 9.0 (`dotnet workload install maui`) |
| CMake | 3.21 |
| Xcode (iOS) | 15 |
| Android NDK (Android) | r25 (API 24+) |

---

## Step 1 — Write your native library

Create your C or C++17 library with a plain-C public API and a single header.

```c
// mylib/include/mylib.h
#ifndef MYLIB_H
#define MYLIB_H

#ifdef __cplusplus
extern "C" {
#endif

float compute(float* samples, int count);

#ifdef __cplusplus
}
#endif

#endif
```

> **Rule:** Keep your public API free of `char*`, function pointers, and
> `long`. Use `int32_t`/`int64_t` for integers, `float`/`double` for
> floating-point. See [architecture.md](architecture.md) for the full type
> mapping table.

---

## Step 2 — Build the native library

### iOS

```bash
./native/scripts/build_ios.sh \
  --target mylib \
  --output artifacts/ios
# Produces: artifacts/ios/libmylib.xcframework
```

You can also invoke CMake directly:

```bash
cmake -S native \
      -B .build/ios/device \
      -DCMAKE_TOOLCHAIN_FILE=native/cmake/ios.toolchain.cmake \
      -DPLATFORM=OS64 \
      -GXcode
cmake --build .build/ios/device --config Release --target mylib
```

### Android

```bash
export ANDROID_NDK_HOME=/path/to/ndk
./native/scripts/build_android.sh \
  --target mylib \
  --output artifacts/android
# Produces: artifacts/android/jniLibs/{arm64-v8a,armeabi-v7a,x86_64,x86}/libmylib.so
```

---

## Step 3 — Generate C# bindings

Use `ferrum-codegen` to turn your header into a `[LibraryImport]` source file:

```bash
# Install as a dotnet tool (once per machine)
dotnet tool install --global Ferrum.Codegen

# Generate bindings
ferrum-codegen \
  --input  mylib/include/mylib.h \
  --output src/MyApp/Interop/MylibBindings.cs \
  --ns     MyApp.Interop \
  --class  MylibBindings \
  --lib    __Internal       # Use __Internal for iOS; adjust for Android
```

The generated file will look like:

```csharp
// <auto-generated/>
// Generated by ferrum-codegen — DO NOT EDIT.

using System.Runtime.InteropServices;

namespace MyApp.Interop;

internal static unsafe partial class MylibBindings
{
    private const string LibraryName = "__Internal";

    /// <summary>Binding for <c>compute</c>.</summary>
    [LibraryImport(LibraryName)]
    internal static partial float compute(float* samples, int count);
}
```

> If `ferrum-codegen` exits with a non-zero code, read the error message.
> It always tells you exactly what construct it cannot bind and why.
> Fix the header or add a hand-written binding for that function.

## Step 4 — Add native references to your MAUI project

In your `.csproj`:

```xml
<!-- iOS XCFramework -->
<ItemGroup Condition="$([MSBuild]::GetTargetFrameworkIdentifier('$(TargetFramework)')) == 'iOS'">
  <NativeReference Include="$(MSBuildThisFileDirectory)..\..\artifacts\ios\libmylib.xcframework">
    <Kind>Framework</Kind>
    <SmartLink>true</SmartLink>
  </NativeReference>
</ItemGroup>

<!-- Android shared libraries -->
<ItemGroup Condition="$([MSBuild]::GetTargetFrameworkIdentifier('$(TargetFramework)')) == 'Android'">
  <AndroidNativeLibrary Include="..\..\artifacts\android\jniLibs\arm64-v8a\libmylib.so" />
  <AndroidNativeLibrary Include="..\..\artifacts\android\jniLibs\armeabi-v7a\libmylib.so" />
  <AndroidNativeLibrary Include="..\..\artifacts\android\jniLibs\x86_64\libmylib.so" />
  <AndroidNativeLibrary Include="..\..\artifacts\android\jniLibs\x86\libmylib.so" />
</ItemGroup>
```

## Step 5 — Call native code from C#

Use `NativeBuffer<T>` for zero-copy buffer passing:

```csharp
using Ferrum.Framework.Buffers;
using MyApp.Interop;

// Allocate and pin a 1024-element float buffer
using var buf = new NativeBuffer<float>(1024);

// Fill via the managed Span<T> — no copy, no boxing
for (int i = 0; i < buf.Length; i++)
    buf.Span[i] = ReadSample(i);

// Pass the raw pointer directly to native code
unsafe
{
    float result = MylibBindings.compute(buf.TypedPointer, buf.Length);
    Console.WriteLine($"Result: {result}");
}
// The pin is released here (Dispose) — GC can compact again
```

### Android library name

On Android the interop constant should be the library name without `lib` prefix
and without `.so` extension:

```csharp
// In your app's interop class, replace __Internal with the library name:
private const string LibraryName = "mylib";  // → libmylib.so
```

Or use `InteropConstants.FerrumLibrary` from `Ferrum.Framework.Interop` for the
test stub, which handles the platform difference automatically.

## Running the MinimalDemo sample

The `samples/MinimalDemo/` project demonstrates the full end-to-end flow using
the `ferrum_add(int, int)` test stub.

```bash
# 1. Build the native test stub
./native/scripts/build_ios.sh      # macOS required
./native/scripts/build_android.sh  # ANDROID_NDK_HOME required

# 2. Uncomment the NativeReference / AndroidNativeLibrary items
#    in samples/MinimalDemo/MinimalDemo.csproj

# 3. Build and run
dotnet build samples/MinimalDemo/MinimalDemo.csproj -f net9.0-ios -c Release
```

## NativeAOT on a real iOS device — manual verification

> **TODO (issue):** Automated CI verification on a physical device is not

> account and provisioning profiles, which cannot be stored in a public
> repository. Track this as a known gap until a self-hosted runner with a
> provisioned device is available.

Until automated CI coverage is in place, run the following steps manually
whenever `NativeBuffer<T>` or the generated bindings change:

### Prerequisites

- macOS with Xcode 15+ and a valid iOS Developer provisioning profile
- An iOS device running iOS 15.0 or later
- .NET SDK 9.0 with the MAUI workload (`dotnet workload install maui`)

### Steps

1. **Build the native fixture:**

   ```bash
   ./native/scripts/build_ios.sh --target ferrum_test_stub
   ./native/scripts/build_ios.sh --target ferrum_dsp_fixture
   ```

2. **Reference the XCFrameworks** in `samples/MinimalDemo/MinimalDemo.csproj`
   (uncomment the `<NativeReference>` items).

3. **Build in Release/NativeAOT mode for a physical device:**

   ```bash
   dotnet build samples/MinimalDemo/MinimalDemo.csproj \
     -f net9.0-ios \
     -c Release \
     -p:RuntimeIdentifier=ios-arm64 \
     -p:_BundlerExecutable=mtouch
   ```

4. **Install and run on the device.**  Open Xcode → Window → Devices and
   Simulators, drag the `.ipa` to your device, and tap it.

5. **Verify the following under NativeAOT** (check Xcode console output):
   - `NativeBuffer<float>` pins memory and returns a non-null `TypedPointer`.
   - `ferrum_add(3, 4)` returns `7`.
   - `ferrum_dsp_scale` operates in-place on a float buffer without crashing.
   - `ferrum_dsp_stats` fills a `FerrumDspStats` struct with correct values.
   - No interpreter fallback messages appear (look for `Falling back to interpreter`
     — there should be none).
   - The app does not throw `MissingRuntimeArtifactException` or any
     reflection-related exception.

6. **Record the result** in the PR description, including the device model,
   iOS version, and .NET SDK version used.

## CMake integration in your own project

If your native library has its own `CMakeLists.txt`, you can use the Ferrum
helper macro to ensure consistent compiler flags:

```cmake
cmake_minimum_required(VERSION 3.21)
project(mylib LANGUAGES C CXX)

# Add Ferrum cmake helpers
list(APPEND CMAKE_MODULE_PATH "${FERRUM_ROOT}/native/cmake")
include(ferrum_helpers)

ferrum_add_static_library(mylib
    SOURCES src/compute.c src/filters.c
    INCLUDES include
)
```

The `ferrum_add_static_library()` macro sets:
- `-fvisibility=hidden` (minimal symbol export surface)
- `-fno-exceptions` / `-fno-rtti` (keeps ABI surface clean)
- `POSITION_INDEPENDENT_CODE ON`
- C11 / C++17 standards
